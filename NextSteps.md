# Next Steps and Feature Ideas

## Priority Roadmap

### 1. Tighten Production Architecture
- Make the ASP.NET API the only public entrypoint in production.
- Keep the Python RAG service internal-only.
- Disable direct frontend-to-Python traffic in production.

### 2. Secure Admin Operations
- Protect admin endpoints such as `/admin/refresh-facts`, `/admin/refresh-facts-background`, and `/admin/refresh-status`.
- Add API key or JWT authentication for refresh operations.
- Restrict CORS and origin access for admin features.

### 3. Add Retrieval and Answer Evaluation
- Automate acceptance criteria checks for retrieval quality.
- Add fixed question sets for leadership, performance, tech stack, achievements, and preferences.
- Validate citation presence and fallback behavior.

### 4. Add Observability
- Log request latency, fallback usage, retrieval distances, and service failures.
- Add a lightweight admin metrics or diagnostics endpoint.
- Track model health separately from API health.

### 5. Containerize the Actual Project
- Add Dockerfiles for the ASP.NET API, Python RAG service, and Angular app.
- Add a Docker Compose file for local orchestration.
- Keep llama.cpp as either a separate service or an external dependency.

## Product Features

### Chat Experience
- Add a conversation list with rename and delete actions.
- Persist multiple chat sessions in a more user-friendly format.
- Add richer source cards with snippet preview, distance, and source metadata.

### RAG Controls in the UI
- Add toggles for `facts-only`, temperature, and retrieval strictness.
- Show why an answer was generated by exposing retrieved FACT blocks.
- Add a visible indicator showing whether the response came from direct RAG or API fallback.

### Knowledge Management
- Add upload or import flow for markdown files into `source_docs/`.
- Add a UI action to trigger fact refresh.
- Show refresh progress and the latest refresh status.

### Multi-Collection Support
- Split knowledge into separate collections such as resume, projects, achievements, and preferences.
- Route queries by intent or let the user choose the knowledge scope.
- Add metadata filters such as date range, section, and project.

### Frontend Structure
- Use Angular routes instead of a single-page shell.
- Add pages for Chat, Sources, Admin, and Settings.
- Add a settings page for session defaults and runtime options.

## Engineering Improvements

### API Hardening
- Add request IDs for traceability.
- Add rate limiting.
- Standardize error responses across ASP.NET and Python services.

### Testing
- Add Python tests for retrieval helpers, indexing flow, and refresh behavior.
- Add Angular tests for chat rendering, loading states, and error handling.
- Add smoke tests covering the full stack.

### Local Developer Experience
- Add `start-all.ps1` and `stop-all.ps1` scripts.
- Add health verification after startup.
- Standardize the Python environment path to avoid venv mismatches.

## Suggested Implementation Order

### Phase 1: Reliability and Safety
1. Secure admin endpoints.
2. Route all frontend traffic through ASP.NET in production.
3. Add startup scripts and health checks.
4. Add structured logging.

### Phase 2: Quality and Operability
1. Add retrieval evaluation scripts.
2. Add smoke tests.
3. Add refresh status visibility in the UI.
4. Add diagnostics and metrics.

### Phase 3: Product Features
1. Add sessions UI.
2. Add source inspection UI.
3. Add settings and admin pages.
4. Add knowledge upload/import workflow.

### Phase 4: Deployment Readiness
1. Add Dockerfiles.
2. Add Docker Compose.
3. Externalize configuration and secrets.
4. Document production deployment steps.

## Concrete Near-Term Tasks
- Add a protected admin refresh page in the Angular app.
- Add a refresh proxy endpoint in ASP.NET instead of calling Python directly from the frontend.
- Add a script that runs index + validation + smoke test in one command.
- Add a source drawer in the chat UI to inspect retrieved facts.
- Add a production config that disables direct RAG API access from the browser.
